// context

为什么需要智能文档审查？

痛点

  • 方案文本动辄 100+ 页（≈50k tokens），人工审查耗时4~6h，遗漏率12%~18%
  • 五大维度交叉检查，人脑覆盖困难
  • 不同行业检查点/术语库差异大，通用规则无法复用
  • 审查结果不统一，不同专家一致性低于70%

核心定位

  • 基于 ReWOO 架构 的智能审查 Agent：Planner-Worker-Solver 解耦
  • 三轮渐进式审查：初审(ReWOO) → 复核(完整性/一致性) → 终审(争议裁决)
  • 支持 9 类文档（可研、标书、申报书等）
  • 行业可插拔：检查点/术语库/政策库按行业分目录
  • 结构化审查报告 + 自动评分评级

// agent-paradigms

智能体模式对比：CoT · ReAct · ReWOO

模式核心思想Token效率工具调用方式推理可观测性适用场景
CoT思维链提示，单次推理生成中间推理步骤⭐ 低（单次LLM）不原生支持工具强（推理步骤显式）数学/逻辑题，无外部依赖
ReActReason+Act交替循环，每步可调用工具获取观察⭐⭐ 中（多轮LLM）交替式，每步结果再喂给LLM强（轨迹可追溯）复杂交互任务，动态检索/计算
ReWOO规划(Planner) → 执行(Worker) → 求解(Solver)，完全解耦⭐⭐⭐ 高（仅2次LLM：规划+求解）批量确定性执行，工具间链式引用(#E1→#E2)，Worker零LLM极强（计划文本+证据链）长文档审查、多工具流水线、成本敏感任务

🔎 为什么选择 ReWOO？

每个检查点需调用3~5个工具，ReAct会引发多轮LLM；ReWOO将工具执行剥离，Worker零成本。
Worker具备跨检查点缓存 + 参数引用，大幅减少冗余计算。
Solver基于完整证据链判定，避免ReAct的上下文膨胀，Token节省约40%~60%。

📊 量化差异（100页文档 / 30检查点）

ReAct：≈90~150次LLM调用，总Token≈400k~800k
ReWOO：≈60次LLM，总Token≈180k~240k，工具可并行
✅ 成本降低约50%，延迟缩减35%，结果确定性更强

ReWOO将“怎么做”与“具体执行”分离，Planner生成工具调用蓝图，Worker批量执行（并行+缓存），Solver综合证据判定——完美匹配文档审查的工具密集+长文本场景。

智能体开发框架选型对比

框架核心思想适用场景优势劣势LLM调用模式
LangChain + LangGraph 链式编排 + 状态图（图结构状态机） 复杂多步智能体、长流程、需精细控制状态 生态丰富（600+集成）、状态持久化、可观测性强、支持循环/分支 学习曲线陡峭、抽象层级多、调试成本较高 灵活：支持ReAct/ReWOO/自定义

Pi-mono 基于“规划-执行-监控”的轻量级Agent框架 单任务规划、低成本原型、嵌入式场景 轻量（～2k行代码）、易扩展、无外部依赖 社区小、文档少、不支持复杂状态图 类似ReWOO但更简化

🎯 DocumentReview 的选型结论

LangGraph 状态图编排完美适配“三轮渐进式审查”（初审→复核→补充检索→终审），每个阶段可单独调试、中断恢复。
生态兼容：LangChain提供现成工具抽象、PDF解析、向量存储，加速开发。
对比Claude SDK：无需依赖特定模型（支持通义千问等国产模型），避免供应商锁定。
对比CrewAI：文档审查是“串行深度推理”而非多角色协作，单Agent + 工具系统更高效。

✅ 最终选择：LangChain Core + LangGraph，兼顾编排灵活性、生态成熟度与多模型兼容性。

框架选型时我们重点评估了四个方向：LangGraph的状态图能清晰表达三轮审查流程，且支持断点续审；Claude SDK虽好但限制模型；CrewAI偏重多角色协作，不适合深度单文档审查；Pi-mono过于简单。因此LangChain+LangGraph是最平衡的选择。

DocumentReview · 技术分享深潜 · ReWOO

deep-dive · rewoo

Planner-Worker-Solver 三阶段

01Planner · 规划器🤖 LLM

输入：检查点指令 + 文档索引摘要（章节标题树，无正文）

输出：计划文本，含 #E1 = ToolName[params] 占位符

关键约束：绝不接触原文，只看章节标题树，避免全文输入浪费 Token

↓
02Worker · 执行器✗ 不调 LLM

输入：Planner 生成的计划文本

输出：所有 #E 占位符替换为真实工具返回结果

关键设计：纯正则解析 + 确定性执行；支持参数间引用；跨检查点共享 tool_cache

↓
03Solver · 求解器🤖 LLM

输入：填充证据后的计划 + 期望输出格式

输出：结构化审查结论 JSON（verdict + evidence + suggestion）

关键设计：递归校验键完整性 + 最多 2 次重试 + 平衡花括号 JSON 提取

“Planner 想，Worker 做，Solver 判”

—— 三者职责完全分离，ReWOO 精髓

Planner只看章节树规划工具调用；Worker纯执行零LLM；Solver基于证据链做最终裁决。

deep-dive · planner

Planner：审查规划的核心

Planner 输入

  • 检查点 instruction
  • 文档索引摘要（章节标题树）
  • 可用工具列表

Planner 输出格式示例

Plan: 首先提取文档开头判断格式规范。然后检查术语使用情况。#E1 = GetDocumentHeader[] #E2 = ExtractTerms[text=#E1] #E3 = MatchTerms[terms=#E2] 推理：从文档开头提取术语，检查是否与术语库一致并做拼写校验。

关键设计要点

不看正文：只看章节标题树，避免全文输入
#E 占位符：工具调用的标准语法
推理指引：告诉 Solver 如何判断
重试机制：最多 2 次，失败加格式提醒

Planner输出计划模板支持预置plan fallback，只生成工具调用蓝图，不处理真实数据。

DocumentReview · 技术分享深潜 · Worker

deep-dive · worker

Worker：确定性执行引擎

Worker 核心逻辑

def execute(plan_text): pattern = r'#E(\d+)\s*=\s*(\w+)\[([^\]]*)]' calls = re.findall(pattern, plan) for id, tool_name, params in calls: resolved = resolve_refs(params, results) cache_key = f"{task_id}:{tool_name}:{json_params}" if cache_key in _cache: result = _cache[cache_key] else: result = tool_registry.run(tool_name, resolved) _cache[cache_key] = result results[f"#E{id}"] = result return fill_evidence(plan_text, results)

关键设计

LLM 调用为零：纯正则 + 确定性执行
参数间引用：#E2 = Tool[params=#E1] 自动解析
跨检查点缓存：同一工具+参数只执行一次
线程安全：细粒度 _cache_lock

Worker纯正则+工具调用，零LLM成本，缓存避免重复执行，参数引用支持链式调用。

DocumentReview · 技术分享深潜 · 归一化

deep-dive · normalization

Verdict 归一化机制

为什么需要归一化？

LLM输出千变万化："有"/"无"/"适用"/"缺失"/"合规"/"不合格" → 必须映射到标准三分类

合格：符合要求
不合格：存在问题
无法判断：证据不足

judgment_type 映射

binary 检查问题：无→合格，有→不合格
presence 检查内容：有→合格，无→不合格
enumeration 枚举型：需人工复核

通过正向/负向词典+类型映射，将所有LLM输出标准化，保证下游一致处理。

DocumentReview · 技术分享深潜 · 终审

deep-dive · final-pass

终审：争议项裁决

触发条件

  • 初审执行失败
  • 复核 confidence "low"
  • 初审与复核 verdict 不一致

无争议则直接归一化

初审与复核一致且置信度高 → 跳过LLM调用，直接 normalize_verdict()

优先使用强模型（config.models.strong）进行终审裁决。

终审按需执行，大部分检查点无需终审，大幅节省Token。

// concurrency

并发安全与资源隔离

任务资源隔离

基于ContextVar实现隐式task_id传递，自动生成UUID，资源按task_id存放。

检查点并行执行

ThreadPoolExecutor并行，max_checkpoints_parallel控制并行度（默认10），每个线程独立实例。

全局锁保护字典操作，ToolRegistry单例+双重检查锁，Worker细粒度缓存锁。

ContextVar保证任务隔离，线程池并行提升吞吐，细粒度锁保证安全。

// persistence

审查结果双重持久化

时间戳文件

output/review_v_{timestamp}_{docHash}.json

完整审查结果+元数据（时间、行业、知识库版本MD5）

累积索引

output/review_results.json

按时间倒序记录每次运行摘要，结构：{"runs": [{version, timestamp, doc_path}]}

容错机制

LLM重试退避，JSON平衡花括号提取，工具缓存自动去重，临时文件自动清理。

双重保存防止数据丢失，知识库版本MD5可追溯。
