为什么需要智能文档审查？

痛点

  • 方案文本动辄 600+ 页（≈1 M tokens），人工审查耗时4~6h，遗漏率20%~30%

  • 支持 9 类文档（可研、标书、申报书等）
  • 行业可插拔：检查点/术语库/政策库按行业分目录

智能体模式对比：CoT · ReAct · ReWOO

模式核心思想Token效率工具调用方式推理可观测性适用场景
CoT思维链提示，单次推理生成中间推理步骤⭐ 低（单次LLM）不原生支持工具强（推理步骤显式）数学/逻辑题，无外部依赖
ReActReason+Act交替循环，每步可调用工具获取观察⭐⭐ 中（多轮LLM）交替式，每步结果再喂给LLM强（轨迹可追溯）复杂交互任务，动态检索/计算
ReWOO规划(Planner) → 执行(Worker) → 求解(Solver)，完全解耦⭐⭐⭐ 高（仅2次LLM：规划+求解）批量确定性执行，工具间链式引用(#E1→#E2)，Worker零LLM极强（计划文本+证据链）长文档审查、多工具流水线、成本敏感任务

🔎 为什么选择 ReWOO？

每个检查点需调用3~5个工具，ReAct会引发多轮LLM；ReWOO将工具执行剥离，Worker零成本。
Worker具备跨检查点缓存 + 参数引用，大幅减少冗余计算。
Solver基于完整证据链判定，避免ReAct的上下文膨胀，Token节省约40%~60%。

📊 量化差异（100页文档 / 30检查点）

ReAct：≈90~150次LLM调用，总Token≈400k~800k
ReWOO：≈60次LLM，总Token≈180k~240k，工具可并行
✅ 成本降低约50%，延迟缩减35%，结果确定性更强

ReWOO将“怎么做”与“具体执行”分离，Planner生成工具调用蓝图，Worker批量执行（并行+缓存），Solver综合证据判定——完美匹配文档审查的工具密集+长文本场景。

🎯 DocumentReview 的选型结论

LangGraph 状态图编排完美适配“三轮渐进式审查”（初审→复核→补充检索→终审），每个阶段可单独调试、中断恢复。
生态兼容：LangChain提供现成工具抽象、PDF解析、向量存储，加速开发。
对比Claude SDK：无需依赖特定模型（支持通义千问等国产模型），避免供应商锁定。
对比CrewAI：文档审查是“串行深度推理”而非多角色协作，单Agent + 工具系统更高效。

✅ 最终选择：LangChain Core + LangGraph，兼顾编排灵活性、生态成熟度与多模型兼容性。

框架选型时我们重点评估了四个方向：LangGraph的状态图能清晰表达三轮审查流程，且支持断点续审；Claude SDK虽好但限制模型；CrewAI偏重多角色协作，不适合深度单文档审查；Pi-mono过于简单。因此LangChain+LangGraph是最平衡的选择。

DocumentReview · 技术分享第三章：ReWOO 深度解析

Planner-Worker-Solver 三阶段深度解析

01Planner · 规划器🤖 LLM

输入：检查点指令 + 文档索引摘要（章节标题树，无正文）

输出：计划文本，含 #E1 = ToolName[params] 占位符

关键约束：绝不接触原文，只看章节标题树，避免全文输入浪费 Token

Plan: 首先提取文档开头判断格式规范。#E1 = GetDocumentHeader[] #E2 = ExtractTerms[text=#E1] 推理：检查术语规范性。
02Worker · 执行器✗ 不调 LLM

输入：Planner 生成的计划文本

输出：所有 #E 占位符替换为真实工具返回结果

关键设计：纯正则解析 + 确定性执行；支持参数间引用；跨检查点共享 tool_cache

pattern = r'#E(\d+)\s*=\s*(\w+)\[([^\]]*)]' for id, tool_name, params in calls: result = tool_registry.run(tool_name, resolve_refs(params))
03Solver · 求解器🤖 LLM

输入：填充证据后的计划 + 期望输出格式

输出：结构化审查结论 JSON（verdict + evidence + suggestion）

关键设计：递归校验键完整性 + 最多 2 次重试 + 平衡花括号 JSON 提取

输出示例：{"verdict":"不合格","evidence":"文档开头'事项'出现8次，其中3次误写为'事顶'","suggestion":"统一替换"}
“Planner 想，Worker 做，Solver 判”

—— 三者职责完全分离，ReWOO 精髓

Planner只看章节树规划工具调用；Worker纯执行零LLM；Solver基于证据链做最终裁决。

DocumentReview · 技术分享第四章：三轮渐进式审查流程-复核（second-pass）

复核：LLM 审查初审质量

审查维度

一致性、完整性、充分性、逻辑性

输出字段

expected_check_items, covered_items, missing_items（具体章节名）

置信度机制

high 可靠
medium 不确定
low → 触发终审

复核不重审文档，而是审查初审结论的质量，missing_items直接驱动补充检索。

DocumentReview · 技术分享第四章：三轮渐进式审查流程-补充检索（supplement）

补充检索：遗漏章节定向分析

触发条件

missing_items 非空 → 对每个遗漏章节提取原文，LLM轻量分析

每个遗漏章节 1 次 LLM 调用，总量 = M 次

合并决策矩阵

初审合格+无新增→合格↑
合格+新问题→不合格
不合格+无新增→维持
无法判断+新发现→不合格

补充检索按需执行，由missing_items驱动，避免冗余LLM调用。

DocumentReview · 技术分享第四章：三轮渐进式审查流程-终审(final-pass)

终审：争议项裁决

触发条件

  • 初审执行失败
  • 复核 confidence "low"
  • 初审与复核 verdict 不一致

无争议则直接归一化

初审与复核一致且置信度高 → 跳过LLM调用，直接 normalize_verdict()

优先使用强模型（config.models.strong）进行终审裁决。

终审按需执行，大部分检查点无需终审，大幅节省Token。

DocumentReview · 技术分享第四章：三轮渐进式审查流程-归一化(normalization)

Verdict 归一化机制

为什么需要归一化？

LLM输出千变万化："有"/"无"/"适用"/"缺失"/"合规"/"不合格" → 必须映射到标准三分类

合格：符合要求
不合格：存在问题
无法判断：证据不足

judgment_type 映射

binary 检查问题：无→合格，有→不合格
presence 检查内容：有→合格，无→不合格
enumeration 枚举型：需人工复核

通过正向/负向词典+类型映射，将所有LLM输出标准化，保证下游一致处理。

工程化能力与部署方案

并发安全与资源隔离

ContextVar隐式task_id，自动UUID，资源按task隔离
ThreadPoolExecutor并行检查点（默认10），ToolRegistry单例+双重检查锁，Worker细粒度缓存锁

Streamlit Web UI

文档上传+行业选择+审查模式切换
后台线程运行，实时日志，支持停止
结果统计面板+表格+详情展开

部署方案

单机版：CLI/Streamlit，个人/小团队
Web A：Docker单容器，5-10人团队
Web B：Docker多实例+Nginx，10-50人
Web C：FastAPI REST API + 自定义前端，50+人/高并发

产出持久化

时间戳文件：output/review_v_{timestamp}_{docHash}.json（含元数据+知识库版本MD5）
累积索引：output/review_results.json，按时间倒序摘要
LLM重试退避，JSON平衡花括号提取，工具缓存自动去重

从单机到企业级弹性扩展，当前运行于单机版；双重持久化+容错机制保障数据可靠。
